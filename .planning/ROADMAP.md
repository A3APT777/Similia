# Roadmap — Similia v1.1 Onboarding

**Milestone:** v1.1 — Onboarding
**Goal:** Провести врача через весь рабочий процесс с первого входа до завершения консультации, включая тур по реперторию.
**Дата создания:** 2026-03-17

---

## Фазы

### Фаза 1 — Верификация UI (VERIFY-01..03)

**Цель:** Аудит всех UI-элементов, на которые будут ссылаться туры этапов 2 и 3. Добавление `data-tour` атрибутов. Проверка соответствия функционала тому, что тур будет описывать.

**Требования:** VERIFY-01, VERIFY-02, VERIFY-03

**Задачи:**
- Пройти по всем шагам планируемых туров и зафиксировать нужные элементы
- Добавить `data-tour` атрибуты: `intake-link-button`, `schedule-button`, `complaints-field`, `mini-repertory-button`, `inline-rx`, `finish-button`
- Добавить `data-tour` атрибуты на странице `/repertory`: `repertory-search`, `rubric-list`, `rubric-grade`, `analysis-panel`, `top-remedies`, `elimination-button`
- Проверить вручную каждый элемент: он существует в DOM, виден пользователю, работает как задумано
- Зафиксировать расхождения (если функционал не соответствует туру — обсудить, не менять молча)

**Критерии успеха (наблюдаемые поведения пользователя):**
1. Разработчик открывает DevTools → видит `data-tour` атрибуты на всех нужных элементах карточки пациента
2. Разработчик открывает DevTools → видит `data-tour` атрибуты на всех нужных элементах страницы реперторий
3. Каждый элемент с `data-tour` кликабелен/интерактивен и ведёт себя так, как описывает тур
4. Список расхождений между функционалом и планом тура задокументирован и согласован

---

### Фаза 2 — Тур этапа 2: карточка пациента (TOUR2-01..04)

**Цель:** Реализовать тур по карточке пациента — предложение продолжить обучение после создания пациента, объяснение IntakeLinkButton и ScheduleButton, переход в редактор консультации.

**Требования:** TOUR2-01, TOUR2-02, TOUR2-03, TOUR2-04

**Задачи:**
- Реализовать триггер тура: после создания пациента (конец этапа 1) показывается предложение продолжить тур
- Добавить шаг тура на IntakeLinkButton с объяснением анкеты для нового пациента
- Добавить шаг тура на ScheduleButton с объяснением создания консультации
- Реализовать переход тура: после записи на приём тур продолжается в редакторе консультации
- Добавить новые шаги в `tour.ts`, новый компонент-триггер или расширить существующий

**Критерии успеха (наблюдаемые поведения пользователя):**
1. Новый пользователь создаёт пациента → видит предложение продолжить обучение и нажимает «Продолжить»
2. Тур подсвечивает кнопку «Пригласить пациента» → пользователь понимает как отправить анкету (по тексту подсказки)
3. Тур подсвечивает кнопку «Записать на приём» → пользователь создаёт консультацию в ходе тура
4. После создания консультации тур автоматически или по клику переводит в редактор консультации

---

### Фаза 3 — Тур этапа 2: редактор консультации (TOUR2-05..10)

**Цель:** Реализовать тур внутри редактора консультации — поле жалоб, мини-реперторий, InlineRx, кнопка завершения приёма, финальный тост.

**Требования:** TOUR2-05, TOUR2-06, TOUR2-07, TOUR2-08, TOUR2-09, TOUR2-10

**Задачи:**
- Добавить шаг тура на поле жалоб с объяснением что туда писать
- Добавить шаг тура на кнопку открытия мини-репертория в тулбаре
- Реализовать цепочку шагов по мини-реперторию: поиск → добавление в анализ → топ препаратов → назначение
- Добавить шаг тура на InlineRx (поле назначения) с уже заполненным препаратом после назначения
- Добавить шаг тура на кнопку «Завершить приём» с объяснением что происходит
- Реализовать тост-поздравление по завершении этапа 2 и предложение перейти к туру по реперторию

**Критерии успеха (наблюдаемые поведения пользователя):**
1. Пользователь видит подсветку поля жалоб → вводит текст симптомов во время тура
2. Пользователь проходит мини-реперторий через тур: находит рубрику, добавляет, видит топ препаратов, назначает
3. После назначения тур показывает InlineRx с заполненным препаратом → пользователь видит результат
4. Пользователь завершает этап 2 → видит тост-поздравление с предложением перейти к туру по реперторию

---

### Фаза 4 — Тур этапа 3: основной реперторий (TOUR3-01..06)

**Цель:** Реализовать полный тур по странице `/repertory` — поиск, рубрики с грейдами, анализ, топ препаратов, элиминация, финальное поздравление.

**Требования:** TOUR3-01, TOUR3-02, TOUR3-03, TOUR3-04, TOUR3-05, TOUR3-06

**Задачи:**
- Реализовать триггер тура на `/repertory`: автозапуск или кнопка-приглашение после этапа 2
- Добавить шаг тура на поле поиска с объяснением поиска на русском и английском
- Добавить шаг тура на список рубрик с объяснением грейдов препаратов
- Добавить шаг тура на кнопку добавления рубрики в анализ и кнопку элиминации (E)
- Добавить шаг тура на панель анализа с топ препаратами и финальным счётом
- Реализовать финальное поздравление — онбординг завершён, `tour_completed` записывается в localStorage

**Критерии успеха (наблюдаемые поведения пользователя):**
1. После завершения этапа 2 пользователь переходит на `/repertory` → тур запускается автоматически или по предложению
2. Пользователь вводит запрос в поле поиска во время тура → видит список рубрик с грейдами
3. Пользователь добавляет рубрику в анализ → тур показывает панель анализа и объясняет топ препаратов
4. По завершении тура пользователь видит финальное поздравление и понимает что онбординг завершён

---

### Фаза 5 — Полировка (переходы, тосты, итоговое тестирование)

**Цель:** Обеспечить плавные переходы между этапами тура, корректность тостов, сквозное тестирование всего онбординга от начала до конца.

**Требования:** Покрывает все REQ-ID в части UX-качества (TOUR2-04, TOUR2-10, TOUR3-01, TOUR3-06 — переходные моменты)

**Задачи:**
- Проверить переход этап 1 → этап 2 (карточка пациента): без разрывов и потери состояния тура
- Проверить переход этап 2 → этап 3 (реперторий): тост корректен, приглашение отображается
- Проверить финальный тост завершения онбординга: корректный текст, localStorage `tour_completed = true`
- Сквозной прогон: новый пользователь проходит все три этапа от начала до конца
- Проверить поведение при прерывании тура (закрыл браузер, вернулся позже)
- Исправить найденные визуальные и логические дефекты

**Критерии успеха (наблюдаемые поведения пользователя):**
1. Новый пользователь проходит этапы 1, 2, 3 без потери контекста тура между переходами
2. Пользователь закрывает и открывает браузер в середине тура → тур возобновляется с нужного места
3. Финальный тост отображается корректно, после него пользователь понимает что обучение завершено
4. Ни один шаг тура не ссылается на несуществующий элемент (нет ошибок Driver.js в консоли)

```
Фаза 1 (Верификация)
    └── Фаза 2 (Тур: карточка пациента)
            └── Фаза 3 (Тур: редактор консультации)
```

Фазы 2–4 последовательны (каждая зависит от предыдущей). Фаза 1 является предусловием для всех остальных.
